# post-merge-retro

## Overview

PR マージ後に、直前の作業を振り返り、**CLAUDE.md / scripts / skills** に改善余地が
あれば提案し、ユーザー承認を得て PR を作成するルーチン。学びを次セッションに
残す運用を、毎マージで確実に回すことが狙い（設計: `docs/superpowers/specs/2026-07-04-post-merge-retro-routine-design.md`）。

**Core principle:** 再帰ガード → 振り返り → 提示して承認 → 承認分だけ実装して PR。

**Announce at start:** 「post-merge-retro スキルで直前のマージを振り返ります。」

## The Process

### Step 1: 再帰ガード

直前にマージされたブランチ名 / PR タイトルを確認する（`git log -1` や、直前の
`gh pr merge` 対象、セッション履歴から分かる範囲でよい）。

- ブランチ名に `post-merge-retro` を含む、または PR タイトルが本ルーチン由来だと
  判断できる場合は、**振り返りを行わず何もせずに終了**する（無限ループ防止）。
- 該当しなければ Step 2 へ進む。

### Step 2: 振り返り

直前にマージした作業（該当 PR / ブランチの diff、このセッションでのやり取り）を
対象に、次の 3 カテゴリで改善候補を洗い出す。各候補は「1 行/項目」の簡潔な形
（`<パターン/コマンド> - <短い説明>`）でまとめる。

- **(a) CLAUDE.md**: 推測が必要だった/不足していた文脈（例: マージ規約、環境の癖、
  gotcha）。
- **(b) scripts**: あれば効率化できた再利用可能なヘルパ（例: 権限付与や定型処理）。
- **(c) skills**: 新規作成・改善した方がよい skill。`skill-creator` プラグインが
  あれば活用してよい。

洗い出しにあたっては、次のような入力を確認する:
- `git log -1 --stat` / マージされた PR の diff
- 直前のセッション内でユーザーから受けた訂正・繰り返し説明・手戻り
- 既存の CLAUDE.md / scripts / skills との重複がないか

### Step 3: 提示と承認

洗い出した候補を (a)/(b)/(c) のカテゴリ別にユーザーへ提示し、どれを採用するか
承認を得る。

```
直前のマージを振り返りました。改善候補:

(a) CLAUDE.md
- <候補1>
- <候補2>

(b) scripts
- <候補>

(c) skills
- <候補>

どれを採用しますか？（全部 / 一部番号 / なし）
```

- **改善点が無ければ「改善なし」とユーザーに報告して終了**（空 PR を作らない）。
- ユーザーが一部のみ承認した場合は、承認された項目のみ Step 4 で実装する。

### Step 4: 実装と PR

承認された変更のみを適用する。

1. ブランチ `chore/post-merge-retro-<topic>` を新規に切る（`<topic>` は今回の
   改善内容を表す短い英語スラッグ）。既に feature ブランチ上にいる場合も、
   main から新規に切り直す。
2. 変更をコミットする（メッセージは日本語、通常のコミットトレーラ規約に従う
   —— `Co-Authored-By` 等を末尾に付与）。
3. `git push -u origin <branch>`。
4. `gh pr create`（base main）。CLAUDE.md「## PR / マージ」の規約に従う
   （件名 `<日本語の説明> (#PR番号)` は squash マージ後に確定するため、
   PR 作成時点では暫定タイトルでよい。対応する issue があれば本文末尾に
   `Closes #n` を付ける）。

完了したら、作成した PR の URL をユーザーに報告する。

## Common Mistakes

**改善候補が無いのに無理に絞り出す**
- **Problem:** 些末な指摘のためだけに PR を作ってノイズになる
- **Fix:** 候補が無ければ素直に「改善なし」で終了する

**承認を待たずに実装してしまう**
- **Problem:** ユーザーが望まない変更が PR に混ざる
- **Fix:** 提示 → 承認 → 実装の順序を必ず守る

**再帰ガードを省略する**
- **Problem:** post-merge-retro 自身の PR マージが次の post-merge-retro を
  呼び、無限にループする
- **Fix:** Step 1 で直前マージが本ルーチン由来かを必ず確認する

## Red Flags

**Never:**
- 承認前に変更を適用する
- 改善候補が無いのに空 PR / 無理な PR を作る
- 再帰ガードをスキップする
- プロダクトコード（`src/`）に手を入れる（本ルーチンは開発プロセス資産のみ対象）

**Always:**
- 振り返りは (a) CLAUDE.md (b) scripts (c) skills の 3 カテゴリで行う
- 候補は 1 行/項目で簡潔にまとめる
- 承認された分だけを PR にする
